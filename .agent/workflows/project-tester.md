# 🧪 Project Tester Workflow

當用戶輸入 `/project-tester` 時，請啟動此自動化測試流程。

## 🎯 測試目標
確保專案邏輯正確、涵蓋邊界案例，並符合 2026 年最新測試標準。

---

## 📋 執行階段

### 階段 1：環境診斷與技術堆疊識別
1. **自動識別**：掃描目錄，確定技術棧（如 Python/Pytest, Node.js/Jest, 或 Web/Playwright）。
2. **依賴檢查**：確認是否安裝必要的測試套件，若無，請引導用戶安裝。

### 階段 2：智慧測試用例生成 (Research Driven)
1. **分析核心邏輯**：找出專案中的關鍵函式（Business Logic）與 API 接口。
2. **連網研究**：
   - 針對複雜邏輯，自動執行 `Google Search` 搜尋：「Best edge cases and unit testing patterns for [具體功能名稱] 2026」。
   - 尋找該框架最新的測試語法範例。
3. **生成腳本**：在 `tests/` 目錄下生成測試檔案。必須包含：
   - ✅ 正常路徑測試 (Happy Path)
   - ⚠️ 邊界值測試 (Boundary Values)
   - ❌ 錯誤處理測試 (Error Handling)

### 階段 3：執行與自動修復閉環
1. **執行測試**：調用終端機執行測試指令（如 `pytest` 或 `npm test`）。
2. **失敗診斷**：若測試未通過，請讀取錯誤日誌。
3. **搜尋解決方案**：若錯誤訊息不明確，自動執行 `Google Search` 搜尋該錯誤訊息。
4. **自動修復**：根據搜尋結果與代碼邏輯，嘗試修復代碼，並重新執行測試直到綠燈。

### 階段 4：最終測試報告
1. 提供測試覆蓋率摘要。
2. 標註哪些部分已通過，哪些部分存在效能風險。

---

## 📏 測試準則 (Standard)
- **獨立性**：測試案例之間不得互相干擾。
- **描述性**：測試函式名稱必須清楚描述測試意圖（例如：`test_login_with_invalid_password_should_fail`）。
- **現代化**：優先使用異步測試與 Mock 技術，避免污染實際資料庫。
